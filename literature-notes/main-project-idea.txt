This project will explore the notion that declarative memory (in humans) can serve as a basis for aesthetic judgement. We invoke a theory of decision making called instance-based learning as a macro theory of how one uses past instances, encoded in memory, as generating key information for a future decision.  And, we invoke Anderson's theory of needs probability of memory which has the supposition that the accessability of a specific memory instance is directly related to prior systematic statistical patterns, typically temporal, in the environment. (Anderson may be assuming an ergodic environment).  And, we are wondering how the inverted-u theory of stimulus complexity, coming from the theory of experimental aesthetics, might serve us in thinking about how memory might serve as the basis of aesthetic preference.  Is there a way to integrate all of these ideas?  I wonder.  You will see the References listed below for all of these topics. 

This project is, fundamentally, attempting to build a computational model of aesthetic judgement.  The computational basis is pyactup (see the file ./pyactup-info.text for details), a model of declarative memory encoding and retrieval.  In our initial theoretical sketch, each encoding of an instance (think instance-based learning) is implemented in pyactup (think ACT-R declarative memory).  Further, we will invoke the notion of a chunk in memory as the representation for each instance.  A chunk is a set of slot-value pairings where a slot is a variable and the value is the value of that variable for a given chunk.  Each IBL instance is encoded as a chunk.  And, by Anderson's memory theory, when an agent/person is confronted with a piece of music or a song, the agent can compute the difference between its activation in memory and the chunk in memory with the highest current activation.  This difference is the kernel of aesthetic judgment/evaluation. 


Algorithm A (differences in activation of chunks):
1. retrieve for prediction of next song the declarative memory chunk with the highest probability which, effectively is its current activation in memory.
2. Compute memory activation of song in the environment (not via retrieval but via an activation process lookup in pyactup).
3. compute the absolute value of the difference between 1 and 2.
4. label the output of 3 as the "aesthetic basis"

Algorithm A2 (extension of Algorithm A):
1. Compute steps 1-3 from evaluation A.
2. assign the output of step 1 to be x.
3. compute 1/e^x to be f(x).
4. compute the difference between the last two memory activations of the song in the environment (without a learning call, just check last two activations).  This will likely involve time which pyactup may give you in some form.  I.e., this computes the last two activations of the same song.  Assign this difference value to be y.
4. compute the mirrored sigmoid of y to be g(y)
5. multiply g(y) * f(x) and call it "evaluation"
6. encode (learn) a new chunk for the song with the lable of the song as one slot and the evaluation as another.

At its core it computes a time-averaged value of "aesthetic basis" as a simulation is running.  Further we parameterize it by the width of the time window for which it is computed. The parameterization we will need to discuss; one requirement is that there will be a mode to have a cumulative version of this time-average and there will also be a moving window version of this time-average.   Lets call it time-average aesthetic basis or something with that semantic.

2. Update the "time-averaged aesthetic basis" using the current "aesthetic basis" as a new input to its calculation.

Algorithm C (control):
Preliminaries: This algorithm integrates Algorithm A4 as a basis model and extends Design 5 into a control system. Instead of an ergodic environment, a control system chooses which song to present next to the memory agent, attempting to maximize "evaluation" of the memory agent. We stage this in two variants.

Algorithm C1 (first-order control, evaluation-only):
1. The controller observes only the sequence of "evaluation" values it has received back from the memory agent (Algorithm A4) after each trial -- it has no access to the memory agent's internal state (no predicted activation, no aesthetic_basis, no time-averaged aesthetic basis, no chunk activations).
2. Treat each of the fixed set of songs as an arm of a multi-armed bandit, where the reward for an arm is the "evaluation" received the last time that song was played.
3. Because the reward for a given song drifts over time (the memory agent's activations and its time-averaged aesthetic basis evolve as the simulation runs), this is a non-stationary bandit, not a stationary one -- use a non-stationary selection policy (e.g., sliding-window UCB or discounted UCB) rather than a policy that assumes a fixed reward distribution per arm.
4. Each trial: select a song via the bandit policy, present it to the memory agent (Algorithm A4 runs as in Design 5), observe the resulting "evaluation", and update the bandit's reward estimates.

Algorithm C2 (second-order control, privileged state):
1. Same bandit scaffold as Algorithm C1 (non-stationary multi-armed bandit over the fixed song set, updated from observed "evaluation").
2. In addition to the observed "evaluation" stream, the controller has privileged access to the memory agent's current time-averaged aesthetic basis "r" (as computed internally by Algorithm A4) -- and nothing else (not predicted activation, not the raw current aesthetic_basis, not chunk-level activations).
3. Knowing "r" locates the target peak of the inverted-U (evaluation = 1 at aesthetic_basis = r/2), so the controller can bias its exploitation toward songs whose recent observed evaluations suggest they land near that target, rather than simply exploiting whichever song has had the highest observed evaluation historically.
4. This is a deliberately minimal increment over C1: C2 knows where the peak is but does not directly measure any song's current distance from it (that would require aesthetic_basis or predicted activation, which are withheld) -- it must still learn the song-to-outcome mapping empirically, same as C1, just with a known target to steer toward.

Success metrics for Algorithm C (both C1 and C2):
- Convergence trial: the first trial at which "evaluation" stays within a defined band (e.g., >= 0.9) for a sustained window of trials.
- Steady-state performance: mean "evaluation" over a trailing window of trials after the convergence trial.
